# API 조합 발견 워크플로우 시스템

## 핵심 아이디어

개별 API 문서를 검색하는 것을 넘어, 여러 API를 논리 노드로 연결해 단일 문서로는 알 수 없는 새로운 활용 방안을 도출한다.

```
기존: 개별 API 문서 검색 -> "이 API로 뭘 할 수 있다"
목표: API A + API B + 논리 노드 -> "이 둘을 조합하면 서비스 X가 가능하다"
```


## 노드 구성

### API 문서 노드 (Source)

각 API 문서가 갖는 정보를 구조화해 노드로 표현한다.

- 제공 데이터 타입 (위치, 시간, 금액, 상태 등)
- 입력 파라미터 목록
- 출력 필드명 + 타입
- 제공기관 / 도메인


### 논리 노드 (Logic)

| 노드 | 역할 |
|------|------|
| Merge | 두 API의 출력을 하나의 컨텍스트로 합침 |
| Join | 공통 키(지역코드, 날짜 등)로 두 API 결합 |
| Filter | 특정 필드 조건으로 교집합 추출 |
| If | 조건 분기 (점수 기준, 필드 존재 여부 등) |
| Semantic Search | FAISS 기반 유사 API 탐색 |


### 분석 노드 (Analysis)

| 노드 | 역할 |
|------|------|
| LLM Analysis | 입력된 API 스키마 조합을 보고 활용 방안 생성 |
| Schema Diff | 두 API의 필드 구조 차이/공통점 비교 |
| Domain Cluster | 이종 도메인 API 간 연결 가능성 평가 |


## 실행 흐름 예시

```
[관광지 API] ---+
                +--> [Merge] --> [LLM Analysis] --> [Export]
[대중교통 API] -+
[날씨 API] -----+
```

LLM이 받는 컨텍스트:
```
API 1: 관광지 정보 (위치, 운영시간, 카테고리)
API 2: 대중교통 노선 (정류장, 배차간격, 도착예정)
API 3: 기상 예보 (기온, 강수확률, 풍속)
```

LLM 출력:
```
날씨와 교통 상황을 반영한 실시간 관광 코스 추천 서비스가 가능합니다.
우천 시 실내 관광지를 자동으로 우선 노출하고, 현재 위치에서 배차 간격이
짧은 노선으로 이동 가능한 동선을 제안할 수 있습니다.
```


## 구현 가능성 평가

| 구성 요소 | 난이도 | 현재 자산 |
|-----------|--------|-----------|
| API 문서 노드화 | 낮음 | JSON 파싱 + 임베딩 완료 |
| 유사 API 탐색 | 낮음 | FAISS 검색 완성 |
| 스키마 필드 추출 | 중간 | `_extract_metadata` 기반 확장 |
| Merge / Join 논리 | 중간 | 프론트 + 백엔드 구현 필요 |
| LLM 조합 분석 | 중간 | 프롬프트 설계가 관건 |
| 새로운 아이디어 생성 | 높음 | LLM 품질 + 컨텍스트 품질에 달려 있음 |


## 핵심 제약

LLM이 자명한 조합만 반복할 위험이 있다. 진짜 새로운 발견이 되려면 **이종 도메인 API 간 연결**이 핵심이다.

- 나쁜 예: 관광지 API + 관광 코스 API (같은 도메인, 자명한 조합)
- 좋은 예: 복지 수급자 API + 의료기관 API + 교통 노선 API -> "거동 불편 노인 맞춤 병원 이동 지원 서비스"

이 차이는 각 API 문서 어디에도 명시되지 않는다.


## 구현 순서

### 1단계: PoC (1~2일)

Merge 노드 1개 + LLM 노드 1개로 가장 빠르게 가치 검증.

- 두 API 문서의 필드 목록을 하나의 프롬프트로 합쳐 LLM 호출
- "이 두 API를 조합하면 어떤 서비스가 가능한가?" 고정 프롬프트
- 결과 품질로 진행 여부 판단

### 2단계: 논리 노드 확장 (1~2주)

- Filter, If, Join 노드 구현
- 노드 실행 상태 UI (idle / running / success / error)
- 노드 간 데이터 전달 구조 확립

### 3단계: 실행 엔진 (2~3주)

- DAG 위상 정렬 기반 순차 실행
- 백엔드 실행 API (`POST /workflow/run`)
- 실행 이력 저장


## 가치의 원천

시스템의 가치 70%는 노드 구조보다 **LLM에게 넘기는 컨텍스트의 품질**에 달려 있다.

- API 필드명 그대로 넘기는 것보다 의미 단위로 정제한 스키마를 넘겨야 함
- 도메인 레이블(복지, 교통, 의료 등)을 함께 넣으면 이종 조합 발견율이 높아짐
- 시맨틱 임베딩으로 미리 이종 도메인 후보를 좁혀서 LLM 입력 크기를 줄이는 것이 효율적
