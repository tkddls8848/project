공공 API 에이전트 인프라 — 독립 서브프로젝트 계획 및 통합 규칙 v0.2
작성일: 2026-05-23 (v0.2 개정)

0. 전체 전략
초기에는 별도 프로젝트 4개로 개발하고, 중기에는 공통 ID·스키마·출처 체계로 연결하며, 후기에는 별도의 통합 레이어 프로젝트(P5) 를 두어 하나의 공공 API 에이전트 인프라로 병합한다.
분리할 프로젝트:

P1. Civic Scenario Catalog — 생활 사건 단위 민원 시나리오 데이터셋
P2. GovAPI MCP Server — 공공 API를 에이전트가 호출 가능한 MCP 도구로 변환
P3. Gov Service Graph — 부처·기관·서비스·API 관계 그래프
P4. Gov24 Link Resolver — CappBizCD 기반 정부24 딥링크·메타데이터 엔진
P5. Integration Layer (신규) — 위 4개 프로젝트 산출물을 결합한 통합 데모·API·UI

최종 통합 제품 정의:
자연어 민원 의도를 받아 시나리오를 찾고, 관련 공공서비스·API·정부24 링크·기관 관계를 조합해 에이전트와 사용자에게 제공하는 공공 API 실행 준비 인프라.

4-B. P5 — Integration Layer (신규)
4-B.1 목적
P1~P4의 산출물을 결합해 자연어 질의 한 줄에서 시나리오·MCP 호출·그래프·딥링크가 모두 응답되는 통합 데모를 만든다. 통합은 그 자체로 별도 노력이 필요한 작업이며, 4개 프로젝트 안에 끼워넣지 않는다.
4-B.2 범위
In Scope

P1 시나리오 카탈로그를 P2 Scenario Tool로 라우팅
P4 딥링크를 시나리오 단계에 주입
P3 그래프 결과를 응답에 첨부
자연어 질의 → 시나리오 식별 → 통합 응답 생성

Out of Scope

새로운 시나리오·API·링크 생성 (각 원천 프로젝트의 책임)
본인인증·신청 자동화

4-B.3 산출물

integration_api/ — 통합 응답 API
demo_ui/ — startup_cafe 데모 화면
integration_tests/ — 4개 프로젝트 결합 시 회귀 테스트

4-B.4 착수 조건
P1·P2·P3·P4가 각각 MVP 성공 기준을 통과한 뒤 착수. 즉 3개월차 시작 시점.

5. 통합을 위해 반드시 지켜야 할 규칙
기존 규칙 1~10은 유지. 규칙 11 신규 추가.
규칙 11. 프로젝트 간 의존은 단방향이며 정해진 순서를 따른다 (신규)
순환 의존이 생기면 통합이 깨진다. 데이터 의존의 방향은 다음 순서로 고정한다.
P4 (Gov24 Link Resolver)
   ↓
P1 (Civic Scenario Catalog)
   ↓
P2 (GovAPI MCP Server)
   ↓
P3 (Gov Service Graph)  ← sink (다른 셋의 결과만 소비)
   ↓
P5 (Integration Layer)  ← 모든 산출물의 최종 소비자
의존 방향 규칙:

P4는 단독. 정부24 링크·CappBizCD 매핑만 다루며, 다른 프로젝트에 의존하지 않는다.
P1은 P4를 소비. 시나리오 단계에 정부24 딥링크가 필요하면 P4 산출물(gov24_service_metadata.jsonl)을 읽는다. 반대 방향 금지.
P2는 P1과 P4를 소비. Scenario Tool은 P1 시나리오를 읽고, 딥링크는 P4를 읽는다. P2가 자체적으로 시나리오를 정의하지 않는다.
P3는 sink. P1·P2·P4의 데이터를 그래프 노드/관계로 변환만 한다. 원천 데이터를 새로 만들지 않는다.
P5는 모든 산출물의 소비자. P1~P4 어느 것도 P5에 의존하지 않는다.

금지 패턴:

P1이 P2의 MCP 도구 정의를 참조 (역방향)
P3가 P1에 새로운 시나리오를 요청 (sink가 원천에 영향)
P4가 P1 시나리오를 기준으로 링크 큐레이션 (P4는 시나리오와 무관해야 함)
어떤 프로젝트라도 P5의 산출물을 참조

판정 기준:
새로운 데이터 의존 관계가 생길 때마다 위 다이어그램에서 화살표 방향과 일치하는지 확인. 일치하지 않으면 거부.

9. 추천 개발 일정 (개정)
0주차 — 사전 작업 (신규)

0-A.1 라이선스 검토
0-A.2 조회 가능 API 사전 매핑 (startup_cafe 기준)
0-A.3 공통 스키마·ID 규칙 확정

1개월차 — 별도 프로젝트 착수

1주차: 0주차 산출물 리뷰 및 보완
2주차: P4 Gov24 Link Resolver 초안 (의존 순서상 최우선)
3주차: P1 시나리오 카탈로그 초안 (P4 소비)
4주차: P2 MCP Tool 1~2개 (P1·P4 소비) + P3 그래프 스키마 초안

2개월차 — 각 프로젝트 MVP

P4: CappBizCD 링크 100개
P1: 시나리오 5개 완성
P2: MCP tool 5개 (조건: 0-A.2에서 검증된 API 사용)
P3: startup_cafe 그래프

3개월차 — P5 Integration Layer 착수 및 1차 통합 데모
공통 질의 "카페를 창업하려면 무엇을 해야 해?"에 대해:

시나리오 식별: startup_cafe (P1)
단계 출력: 사업자등록, 영업신고, 위생교육, 4대보험 (P1)
관련 기관 출력 (P1)
정부24·기관 딥링크 출력 (P4)
호출 가능한 공공 API 실행 (P2)
기관·서비스 관계 그래프 표시 (P3)

위 6단계를 통합 응답으로 묶는 작업이 P5의 책임.
