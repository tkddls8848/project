# Hermes 에이전트 확장 서비스 탐구

- 문서 상태: 탐구 결과 (구현 승인 전)
- 작성 기준일: 2026-07-20
- 선행 문서: [`hermes_tool_loop_plan.md`](hermes_tool_loop_plan.md)
- 탐구 범위: 저장소 내 전체 하위 프로젝트와 archive, korea100 자산

## 1. 결론 요약

현재 Hermes 오케스트레이션은 Nara MCP 도구 네 개(`search_api_docs`,
`get_api_detail`, `derive_relations`, `compose_service_plan`)로
"검색 → 검토 → 관계 → 계획"만 다룬다. 저장소를 탐구한 결과, **앱의 사용
목적(근거 기반·읽기 전용·계획 작성)을 그대로 지키면서** 이미 저장소 안에
존재하는 자산만으로 추가할 수 있는 에이전트 서비스 후보 일곱 개를 확인했다.

우선순위 권장:

| 순위 | 후보 | 근거 자산 | 새 백엔드 필요 |
|---|---|---|---|
| 1 | 계획 검증 에이전트 (Plan Critic) | 기존 4개 도구 + 최종 JSON 계약 | 없음 |
| 2 | 법령 근거 에이전트 | korea100 law-to-process MVP (`law.go.kr` DRF) | 기존 MVP 재사용 |
| 3 | 문서 신선도 에이전트 | `nara_storage/manifests/` 체크섬 | 없음 (읽기만) |
| 4 | 워크플로우 내보내기 도구 | nara_dashboard flow JSON 계약 | 없음 |
| 5 | 제도 맥락 에이전트 | korea100 제도 187개·절차 노드 1,837개 | 정적 데이터 서빙 |
| 6 | 평가 러너 에이전트 | `evaluation/golden_queries.json` + shadow mode | 오프라인 러너 |
| 7 | 생활사건 링크 에이전트 | archive의 gov24 link resolver 데이터셋 | 데이터셋 재활성화 |

반대로 **실행형 에이전트(archive의 OpenClaw 방향), 브라우저·터미널·메시징
도구 개방은 여전히 부적합**하다. 근거는 8절에 정리한다.

## 2. 앱의 사용 목적 재확인

확장 후보를 판정하는 기준은 기존 문서에서 이미 고정되어 있다.

1. 자연어로 공공 API 문서를 찾고, 상세 근거를 검토한다 (README PoC 범위).
2. 문서 사이 관계를 **도구가 반환한 근거로만** 주장한다 (SKILL.md Safety).
3. 최대 세 개 문서로 **실행이 아닌** 행정 서비스 계획 초안을 만든다.
4. 실제 행정 API 실행·외부 시스템 변경·브라우저 자동화는 범위 밖이다.
5. 모든 선택·제외에 추적 가능한 이유를 남긴다 (tool loop plan §7.2 계약).

따라서 "부합하는 확장"이란 **계획의 근거를 더 두껍게 만들거나, 근거의 품질을
검증하거나, 계획 결과를 사람이 검토하기 좋은 형태로 전달하는 것**이다.
실행 능력을 늘리는 것이 아니다.

## 3. 후보 A — 계획 검증 에이전트 (Plan Critic)

### 3.1 무엇인가

메인 설계 run이 끝난 뒤, **읽기 전용 도구만 가진 별도 Hermes 프로필**이 최종
JSON 계약을 받아 독립적으로 재검증하는 2차 에이전트다. proposer–reviewer
패턴이며, tool loop plan의 평가 지표(근거 없는 관계 주장률 0%, 선택 API 상세
조회율 100%)를 사람 대신 1차 필터링한다.

### 3.2 검증 항목

- `selected_apis`의 모든 `service_id`가 실제 `get_api_detail`로 조회 가능한가
- `relations` 주장이 `derive_relations` 재호출 결과의 부분집합인가
- `plan` 본문이 선택 문서의 입력·출력 필드를 벗어나 필드를 지어내지 않았는가
- `executed: false`와 비실행 문구가 유지되는가
- `rejected_apis`의 제외 이유가 상세 문서 내용과 모순되지 않는가

### 3.3 통합 방식

- 신규 백엔드 불필요. 기존 MCP 도구 네 개를 그대로 쓰되 `compose_service_plan`은
  제외한 세 개만 노출한다 (검증자는 계획을 다시 만들면 안 된다).
- 별도 프로필(`nara-critic`)로 도구 화이트리스트를 분리한다.
- 출력은 `verdict: pass | evidence_gap | contradiction` + 항목별 근거로
  구조화하고, UI에는 경고 배지로만 반영한다. 자동 수정은 하지 않는다.
- 비목표인 "여러 Agent 병렬 운영"과 충돌하지 않도록 **순차 실행**(메인 run
  종료 후 1회)으로 한정한다.

### 3.4 판정

가장 우선 권장. 새 코드가 가장 적고, 단계 5(shadow 평가)의 자동화 기반이
되며, 앱의 핵심 가치인 "지어내지 않기"를 시스템으로 강제한다.

## 4. 후보 B — 법령 근거 에이전트

### 4.1 근거 자산

`korea100/sources/law-to-process-mvp-2026-07-09/`에 이미 실행 가능한 FastAPI
프로토타입이 있다.

- `POST /api/law/search` — 국가법령정보센터(law.go.kr) DRF 법령 검색
- `POST /api/law/detail` — 법령 상세 + 조문 평문화(`flatten_law_text`)
- `POST /api/process/extract` — 조문 텍스트에서 행위주체·행위·문서·기한을
  추출해 스윔레인 절차 모델 생성 (규칙 기반, LLM 아님)

### 4.2 무엇을 하는가

행정 서비스 계획 초안에 **법적 근거 열을 추가**한다. 흐름:

1. 계획 초안이 만들어진 뒤, 선택 API의 소관 제도·업무 키워드로 법령을 검색한다.
2. 후보 법령의 조문을 조회해 해당 서비스의 근거 조문(신청 자격, 처리 주체,
   기한)을 확인한다.
3. 확인된 조문만 `legal_basis` 필드로 계획에 첨부한다. 확인 실패는 korea100
   원칙대로 `unverified`로 표기하고 추정하지 않는다.

### 4.3 신규 MCP 도구와 예산

| 도구 | 용도 | 호출 제한 |
|---|---|---|
| `search_law` | 법령명·키워드 검색 | 최대 3회 |
| `get_law_articles` | 조문 상세 + 평문 | 최대 4회 |
| `extract_process_from_law` | 조문 → 절차 모델 (선택적) | 최대 1회 |

모두 읽기 전용 외부 조회다. law.go.kr는 공개 API이므로 실행·변경 위험이 없다.

### 4.4 판정

앱 목적과의 정합성이 가장 높은 **기능 확장**이다. "이 API 조합으로 서비스를
만들 수 있다"에서 "이 서비스는 어느 법 조문이 근거이고 처리 주체·기한이
무엇이다"로 계획의 신뢰 수준이 올라간다. MVP가 이미 검증돼 있어(NOTES.md)
비용도 낮다. 단, 외부 네트워크 의존이 생기므로 timeout·실패 시 계획 본문은
유지하고 법령 열만 비우는 fail-soft 처리가 필요하다.

## 5. 후보 C — 문서 신선도 에이전트

### 5.1 근거 자산

`nara_crawler`는 `../nara_storage/manifests/{run_id}.json`에 실행별 수집 파일
목록과 **체크섬**을 남기고, 같은 `api_id` 재크롤 시 문서를 덮어쓴다. 한편
`nara_search`의 FAISS 인덱스는 `/build` 시점의 문서로 고정된다. 즉 "인덱스가
아는 문서"와 "저장소의 최신 문서"가 어긋날 수 있는 구조적 틈이 이미 있다.

### 5.2 무엇을 하는가

계획에 사용된 문서의 신선도를 검사해 경고를 첨부한다.

| 도구 | 용도 | 호출 제한 |
|---|---|---|
| `check_doc_freshness` | 선택 service_id의 manifest 체크섬 vs 인덱스 빌드 시점 비교 | 최대 1회 (배치) |

출력 예: "15000827은 마지막 인덱스 빌드 이후 재크롤되어 내용이 변경되었을 수
있음 → 상세 조회 결과 기준으로 판단했음을 명시". 재크롤 실행이나 인덱스
재빌드 **트리거는 포함하지 않는다** (쓰기 작업이므로 사람이 수행).

### 5.3 판정

작지만 실질적인 근거 품질 개선. manifest 읽기만 하므로 위험이 거의 없다.
`insufficient_evidence` 판정의 새로운 사유("문서가 오래됨")로 자연스럽게
연결된다.

## 6. 후보 D — 워크플로우 내보내기 도구

### 6.1 근거 자산

`nara_dashboard`는 `{format: "nara-dashboard-flow", version: 1, nodes[],
edges[]}` 계약의 flow JSON 가져오기를 지원하고, 관계 엣지의 근거 검토·승인
UI를 이미 갖고 있다.

### 6.2 무엇을 하는가

에이전트 run의 최종 결과(선택 API 노드 + 확인된 관계 엣지)를 대시보드
flow JSON으로 변환해 다운로드로 제공한다. 사람이 React Flow 캔버스에서
에이전트의 결과를 이어서 검토·편집하는 **사람-검토 루프**가 완성된다.

- Hermes에 노출하는 MCP 도구가 아니라 **PoC 백엔드의 결과 후처리 엔드포인트**
  (`GET /agent/design-runs/{id}/flow`)로 충분하다. 도구 예산에 영향이 없다.
- 대시보드 README가 명시한 한계(관계 메타데이터 미직렬화)가 있으므로, 근거
  텍스트는 노드 설명 필드에 넣는 우회가 필요하다.

### 6.3 판정

에이전트 기능 자체보다 **결과 소비 경로** 확장이다. 구현이 하루 단위로 작고
기존 UI 자산을 재사용하므로 가성비가 높다.

## 7. 후보 E·F·G — 중기 후보

### 7.1 제도 맥락 에이전트 (korea100)

korea100은 제도 187개를 9칸 캔버스와 스윔레인(절차 노드 1,837개, 법령 인용
2,743건)으로 구조화한 **정적 검증 데이터**다. `search_institutions` ·
`get_institution_canvas` 도구를 추가하면, 계획 초안에 "이 API 조합이 기존
제도의 어느 단계(레인·행위주체)에 꽂히는지, 알려진 병목이 무엇인지"라는
제도 맥락을 붙일 수 있다. 정적 JSON 서빙이라 위험은 낮지만, korea100 데이터
계약을 PoC가 소비하는 로더를 새로 만들어야 하므로 후보 B보다 비용이 크다.
법령 근거 에이전트(B)가 자리 잡은 뒤 같은 결과 계약에 얹는 것을 권장한다.

### 7.2 평가 러너 에이전트

tool loop plan 단계 5는 골든 질문 20개 이상으로 baseline/agent 비교 보고서를
요구한다. 현재 `golden_queries.json`은 5개다. 오프라인에서 골든셋을 순회하며
두 모드를 실행하고 지표 표를 생성하는 러너는 tool loop plan §5.2가 이미
"Python Library는 오프라인 평가 러너에서만 검토"라고 남겨둔 자리에 정확히
들어맞는다. 후보 A(검증 에이전트)의 verdict를 지표로 재사용하면 사람 평가
부담이 줄어든다.

### 7.3 생활사건 링크 에이전트

archive의 gov24 link resolver는 "상황(생활 사건) → 정부24/기관 링크"를 사람이
큐레이션한 JSONL 데이터셋 프로젝트다. 계획 초안 말미에 "실제 신청은 여기서"
링크를 첨부하는 `resolve_service_links` 도구로 쓸 수 있다. **사람이 검증한
링크만** 반환하므로 근거 원칙에 부합하지만, 데이터셋이 archive 상태라
큐레이션 재가동이 선행돼야 한다. 장기 후보로 분류한다.

## 8. 부적합 판정

탐구 중 확인했으나 **추가하지 않아야 하는** 방향:

| 후보 | 부적합 이유 |
|---|---|
| 실행 에이전트 (OpenClaw 재활성화) | Combiner·PoC README가 실행·승인·감사를 명시적으로 범위 밖으로 선언. archive 보류는 의도된 결정 |
| 브라우저·터미널·파일 쓰기 도구 | tool loop plan §3.2 비목표, §12 보안 설정과 정면 충돌 |
| 재크롤·인덱스 재빌드 자동 트리거 | 쓰기 작업. 신선도 에이전트는 보고까지만 (5.2절) |
| 무승인 메모리·스킬 학습 에이전트 | 단계 6에서 shadow 평가 통과 후, 승인형으로만 허용 |
| 여러 에이전트 동시 병렬 실행 | 비목표 명시. 검증 에이전트도 순차 1회로 한정 (3.3절) |

## 9. 권장 진행 순서

1. **후보 A (계획 검증)** — 신규 백엔드 없음. `nara-critic` 프로필 + verdict
   계약 + UI 배지. 단계 5 평가의 자동 필터가 된다.
2. **후보 D (flow 내보내기)** — PoC 엔드포인트 하나. 사람 검토 루프 완성.
3. **후보 C (신선도 검사)** — manifest 읽기 도구 하나. 경고 계약 확장.
4. **후보 B (법령 근거)** — law MVP를 사이드카로 실행, MCP 도구 3개 추가.
   결과 계약에 `legal_basis[]` 필드 추가.
5. 이후 후보 E → G → F 순으로 재평가.

공통 원칙은 tool loop plan을 그대로 상속한다: 새 도구는 모두
`tools.include` 화이트리스트에 명시하고, 도구별 호출 예산을 정의하며,
확인 실패 시 지어내지 않고 `unverified`/`insufficient_evidence`로 종료한다.
