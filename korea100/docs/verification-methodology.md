# 업무구조도 검증 방법론

101개에서 시작해 현재 216개로 확장된 제도의 캔버스·업무구조도에서 내용 오류를 체계적으로 찾아 정정하기 위해 설계한 방법론. 2026-07-12 하루 동안 전 층위를 실전 적용해 인용·그래프·레인 오류 238건과 L2 정정 약 130건을 처리하며 검증됐다(PR #27, #30, #39~#44).

## 0. 전제 — "검증됨"의 층위

조문 번호의 **존재** 확인(기존 verify-articles)과 그 조문이 해당 업무의 **행위·주체·기한을 실제로 규정하는지**(의미 검증)는 다른 문제다. 이 방법론은 의미 층위까지를 다룬다.

## 1. 오류 유형학 — 무엇이 틀릴 수 있는가

| 유형 | 정의 | 실측 사례 | 잡는 층위 |
|---|---|---|---|
| A. 형식 결함 | article 필드에 조문 번호가 아닌 문구 | "주민조례청구", "~ 관련 조문" (서술형 210건) | L0 (정규식) |
| B. 주체(레인) 오류 | 행위 주체가 다른 기관으로 표기 | 기초생활보장 이의신청 심사: 위원회→시·도지사(§39) | L1·L2 |
| C. 조문-행위 불일치 | 조문은 실재하나 그 행위를 규정하지 않음 | 입법예고 근거가 지방자치법 §28(행절법 §41이 맞음) | L1·L2 |
| D. 기한 오류·미반영 | 법정 기한 누락·오기·개정 미반영 | 재건축 동의율 3/4→70%(2025-01 개정) | L2 |
| E. 필수 단계·권리 누락 | 법정 단계가 도식에 없음 | 이의신청·심의회·사전동의·조서 열람 — **최다 반복 패턴** | L2·L3 |
| F. 허구 단계·지어낸 기한 | 법령에 없는 것을 그럴듯하게 표기 | "내부 목표 60일", 비법정 심의회를 법정처럼 | L2 |
| G. 부처·기관명 개정 미반영 | 정부조직 개편 등 | 기획재정부→기획예산처·재정경제부 (170건) | L0+현행 대조 |

## 2. 5단계 깔때기 — 싼 것부터, 전수에서 정밀로

### L0. 기계 감사 (비용 0, 전수) — `web/scripts/audit-process.mjs`
- 인용 형식(서술형/묶음), 그래프 도달성(고아·도달불가·종결없음), 역방향 sequence, 빈 레인·스테이지, 제도 간 통계 이상치(z>2)
- 결과는 `docs/audits/process-audit-<날짜>.json` 우선순위 큐
- 인정 인용 형식: `제N조(의M)` / 부칙 / 별표 / `조약 제N호` / `일련번호 N` / `예규·고시·훈령 제N호`
- `unverified: true` 플래그(확인 불가 문서화분)는 감사 제외

### L1. 조문 의미 대조 (노드 단위) — `docs/recipes/l1-citation-verification.md`
법령 목차 조회 → 노드의 행위로 조문 매핑 → 모호하면 원문 확인 → 묶음 분리. **지어내기 절대 금지**: 확인 못 한 조문은 인용하지 않고 `unverified` + fieldVerification 이관.

### L2. 블라인드 재구성 (제도 단위, 가장 강력) — `docs/recipes/l2-blind-reconstruction.md`
검증자가 **기존 도식을 보지 않고** 법령 원문만으로 절차를 재구성한 뒤 diff. 자기 생성물 검토의 확증편향을 깨는 유일한 방법으로, D(최신 개정)·F(허구)형을 잡는 주력. 부산물로 "법의 침묵"(법령이 규정하지 않는 지점) 수집 — 콘텐츠 소재.

### L3. 페르소나 워크스루 (당사자 시점)
구체 사례의 당사자(신청인·이의신청인·환자·근로자)가 도식을 따라 걸을 때 막히는 지점 탐색. **빈 당사자 레인은 E형 오류의 직접 신호**. 이 프로젝트의 확립 원칙: 당사자 레인에 신청·진술·불복 등 행위 노드 최소 2개.

### L4. 외부 그라운드트루스 대조 (표본)
정부24 절차 안내, 부처 업무편람, 판례·해석례(`prec`·`expc` 타깃), 위원회 결정문과 대조. 법령으로 안 보이는 운영 현실은 fieldVerification으로.

## 3. 판정 원칙 (L2·L3 공통)

- **고치는 것**: C(근거 오인용), E(필수 단계·당사자 권리 누락 — 노드 추가는 status done/waiting, current 불변), D(기한), F(비법정 기구는 unverified 표기)
- **고치지 않는 것**: 모델링 압축, 레인 묶음, 명명 스타일 — 취향은 오류가 아니다
- **불복·이의·소청 노드는 반드시 심사·결정 노드를 경유해 원처분으로 회귀** (validator 규칙)
- 정정은 `process.warnings`에 날짜·근거·대조 식별자(MST 등)와 함께 기록. 이력은 절대 소급 수정하지 않는다

## 4. 원문 검증 인프라 — 법제처 DRF API

타깃 지도·사용법은 `docs/development.md` 참조. 요약: `law`(법령) / `admrul`(행정규칙 — 훈령·예규·고시·지침) / `ordin`(자치법규) / `trty`(조약) / `pi`(공단·공공기관 규정) / `prec`·`expc`(판례·해석례). MCP 장애 시에도 curl 직접 조회 가능(OC=test).

- **조례 위임 사항**: 예시 지자체 하나를 골라 원문 검증하고 "— 예시" + "지자체별 상이" 명시 (예시는 되도록 서울 제외 — 부산·경기 우선)
- **admrul·pi에도 없는 내부 문서**: "미등재 확인(날짜)" 표기 — **부재의 확인도 검증이다**

## 5. 오류의 기계화 — 방법론이 스스로 자라는 구조

L1~L4에서 오류 유형이 확정될 때마다 묻는다: "이걸 L0 규칙으로 내릴 수 있는가?" 실례: 서술형 인용 발견→정규식 규칙, 조약 인용→`조약 제N호` 형식 인정, 발령번호 인용→형식 인정, 임용령·징계령이 하위법령 집계에서 빠짐→규칙 확장. 신규 제도가 추가되면 감사가 즉시 같은 기준으로 걸러낸다(상설 게이트).

정정 완료 사례들은 골든셋(회귀 기준)이 된다 — 같은 오류가 재발하면 감사·리뷰에서 걸려야 정상.
