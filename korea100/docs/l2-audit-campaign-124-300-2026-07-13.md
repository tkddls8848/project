## 1. 감사 설계

### 1.1 깔때기

1. **L0 기계 감사** (`web/scripts/audit-process.mjs`, 전수·비용 0): 대상 139개 중 130개에서 261건 검출. 실질 판정 대상은 backward-sequence 22건 + empty-lane 약 50건. 서술형·묶음 인용(A형)은 0건이어서 L1(조문 의미 대조)은 생략하고 L2로 직행.
2. **L2 블라인드 재구성** (제도 단위, 주력): 검증 에이전트가 기존 도식을 열지 않은 채 법제처 DRF API 원문만으로 절차 골격을 재구성(`/tmp/l2-recon-<slug>.json`)한 뒤 기존 도식과 diff. 확증편향을 차단하는 유일한 방법이며 D(기한)·F(허구)형을 잡는 주력.
3. **판정 원칙**: 고치는 것 = C(근거 오인용)·E(필수 단계·당사자 권리 누락)·D(기한)·F(비법정 기구의 법정 표기). 고치지 않는 것 = 모델링 압축·레인 묶음·명명 스타일. 지어내기 절대 금지 — 원문 미확인 조문은 인용 불가, `unverified: true` + fieldVerification 이관.

### 1.2 L0 발견의 에이전트 위임

L0의 backward-sequence(역방향 엣지)와 empty-lane(빈 레인) 발견을 해당 배치 에이전트에 넘겨 법령 근거로 판정시켰다:
- **backward-sequence 22건** → 대부분 반려·보완 회귀로 확정되어 `loop` 타입 전환. 예외: 스테이지 오배치 2건(재정렬), 의도적 역방향 3건(§2.4).
- **empty-lane** → 레인 주체의 법정 행위가 있으면 노드 추가(E형), 없으면 정당한 모델링으로 존치. 삭제는 금지.

## 4. 운영 기록 (병렬 27 에이전트)

1. **광역 `git add` 사고 재발**: L2 레시피의 `git add web/data/ docs/`가 병렬 에이전트의 미커밋 파일을 휩쓸어 커밋 귀속 혼입 5건 발생(데이터는 무결). 배치 15부터 명시 경로 add로 오버라이드해 차단. **레시피 원문 수정 필요(§5).**
2. **최종 보고 유실 빈발**: 27개 중 ~10개 에이전트의 최종 보고가 "Complete."로만 회수 → 전 배치를 커밋 로그 + 파일 내 warnings 레코드 실측으로 검수(누락 0 확인). 전 세션 교훈("보고는 유실된다, 실측하라")이 그대로 유효.
3. **처리량**: 배치(제도 7개)당 평균 20~26분, 토큰 12만~28만. 동시 7슬롯이 안정적 (index.lock 충돌은 2초 재시도로 전량 해소).
4. **L0 동일 지문 반복 = 템플릿 탐지 신호**: 방법론 §5(오류의 기계화)에 편입 후보 — "N개 파일이 동일 rule+edge+position 발견을 공유하면 일괄 생성 의심" 규칙.
